Claude Code silently committed my Contentful Management Token to git history through an MCP config file.

GitGuardian caught what Claude Code didn't. Here's exactly how an MCP config file caused it and what I do differently now.

By Nika Karliuchenko · 6 min read · March 7, 2026
$3.14 Session cost
488K Total tokens
opus-4-6 Model

I was setting up the Contentful MCP server for the first time. Claude Code walked me through the configuration, and I followed along — pasting my Management Token into a config file as instructed. It worked. Claude Code connected to Contentful, listed my content types, built six of them in under ten minutes. I was impressed.

Two hours later, GitGuardian sent me an email. My Contentful Management Token was in my git history. Not in a .env file. Not in a file I'd added to .gitignore. In a .claude/mcp.json config file that Claude Code had created — and that I had committed without thinking.

The thing that got me wasn't carelessness. It was trust. I trusted the tool to handle secrets the way I expected.

How it happened

The MCP setup process requires a config file that specifies which MCP servers to connect to, along with their authentication credentials. Here's roughly what mine looked like:

JSON · .claude/mcp.jsonCopy
"mcpServers": {
  "contentful": {
    "command": "npx",
    "args": ["-y", "@contentful/mcp-server"],
    "env": {
      "CONTENTFUL_MANAGEMENT_TOKEN": "CFPAT-xxxxxxxxxxxxxxxxxxxx" ← this went to git
    }
  }
}

The file was in a project directory I was actively working in. I ran git add . and committed. Standard muscle memory. The MCP config file had no warning that it contained credentials. Nothing in the Claude Code workflow flagged it.

What GitGuardian caught

GitGuardian's free tier scans public repos in real-time. The email came fast. By the time I saw it, the commit was already pushed. The token was already compromised — even for a brief window on a low-traffic personal repo, a leaked Management Token gives full write access to your Contentful space.

A Contentful Management Token has full read/write access to your space. It can create, update, and delete content types, entries, and assets. It is not a read-only key.

— Contentful documentation

What I did to fix it

Here's the remediation sequence, in the order I ran it:

  1. Revoke the token immediately in the Contentful web app under Settings → API keys → Content management tokens. Don't wait.
  2. Remove the credential from git history using git filter-branch or BFG Repo Cleaner. Force-push to main.
  3. Add the config file to .gitignore before creating a new token.
  4. Create a new Management Token and update the MCP config with the new value.
  5. Verify the old token is dead by attempting an API call with it.

What to do differently from the start

There are a few patterns that would have prevented this entirely:

  • Never put credentials in config files that live in project directories. Use environment variables referenced from the config, not values inline.
  • Add MCP config directories to .gitignore immediately when you start a project — before you ever touch them.
  • Use a restricted token if your tool supports it. Contentful's Management API tokens are all-or-nothing, but some tools support scoped tokens.
  • Run git diff --staged before every commit. It takes 3 seconds. This is the oldest advice in the book and I still didn't do it.
Worth knowing

The correct pattern for the Contentful MCP config is to reference an environment variable — "CONTENTFUL_MANAGEMENT_TOKEN": "${CONTENTFUL_TOKEN}" — not a literal value. The MCP server documentation doesn't make this obvious up front.

The bigger pattern

This wasn't a one-off mistake. It's a pattern I keep seeing when AI tools generate configuration. The tool optimizes for "it works" — it puts the credential where it needs to be for the process to succeed. The concern of where that file ends up in version control is not part of the tool's evaluation function.

That's not a criticism. It's a constraint worth knowing. When you're working with AI tools that generate config files, credential handling is still your responsibility. The tool will not catch this for you. GitGuardian will — but that's a backstop, not a strategy.
